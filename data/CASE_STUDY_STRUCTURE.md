# Case Study Folder Structure

Each case study in the waterpath-scenario-builder has its own dedicated folder structure in the `data/` directory.

## Folder Naming Convention

Case study folders are named using the pattern:
```
{case_study_name}_{case_study_id_first_8_chars}
```

For example: `my-water-project_a1b2c3d4`

## Folder Structure

Each case study folder contains three subdirectories:

```
data/
├── case-study-name_12345678/
│   ├── input/          # Input files for GloWPa (CSV, R scripts, data files)
│   ├── output/         # Output files generated by GloWPa (results, plots, logs)
│   ├── config/         # Configuration files (YAML, JSON, XML, etc.)
│   └── datapackage.json # Metadata describing the case study and its resources
├── another-study_87654321/
│   ├── input/
│   ├── output/
│   ├── config/
│   └── datapackage.json
└── ...
```

## Metadata Management

Each case study includes a `datapackage.json` file that follows the [Data Package](https://specs.frictionlessdata.io/data-package/) specification. This file contains:

- **Case study metadata**: Name, title, description, creation date, contributors
- **Resource catalog**: Detailed information about all input and config files
- **Data lineage**: Sources, licenses, and provenance information
- **Technical details**: File formats, sizes, schemas for tabular data

### Datapackage.json Structure

```json
{
  "profile": "data-package",
  "name": "nile-basin-analysis",
  "title": "Nile Basin Water Analysis",
  "description": "Comprehensive water analysis for the Nile Basin region",
  "created": "2025-10-15T20:30:00",
  "keywords": ["water management", "glowpa", "waterpath", "case study"],
  "contributors": [{"title": "Your Name", "role": "author"}],
  "resources": [
    {
      "name": "water_data",
      "path": "input/water_data.csv",
      "title": "water_data.csv",
      "format": "csv",
      "profile": "tabular-data-resource",
      "mediatype": "text/csv",
      "bytes": 1024,
      "schema": {}
    }
  ]
}
```

## File Type Organization

When uploading a ZIP file to create a case study, files are automatically organized by type:

- **Input Directory**: `.csv`, `.txt`, `.dat`, `.R`, `.py` files
- **Config Directory**: `.json`, `.yaml`, `.yml`, `.xml`, `.conf` files  
- **Output Directory**: Generated automatically when GloWPa runs

## GloWPa Integration

When you click "Run" on a case study:

1. GloWPa container mounts the specific case study folder
2. Input files are available at `/app/data/{case_study_folder}/input/`
3. Config files are available at `/app/data/{case_study_folder}/config/`
4. Output files are written to `/app/data/{case_study_folder}/output/`

## Backward Compatibility

The default `data/input/`, `data/output/`, and `data/config/` folders are still available for:
- Legacy scripts and workflows
- Testing and development
- Default GloWPa runs without a specific case study

## Usage Tips

1. **Organize by Project**: Create separate case studies for different water management projects
2. **Version Control**: Use meaningful case study names to track different versions
3. **File Management**: Keep input files organized and properly named
4. **Configuration**: Store model parameters and settings in the config folder
5. **Results**: All GloWPa outputs will be saved in the output folder for easy access
6. **Metadata**: Use the 👁️ (eye) button in the web interface to view case study metadata
7. **Documentation**: Update the datapackage.json description field to document your case study

## Metadata Viewing

Click the eye icon (👁️) next to any case study to view its metadata in a popup window showing:

- **Case study details**: Name, description, creation date, contributors
- **File inventory**: Complete list of input and config files with sizes and types
- **Data lineage**: Sources, licenses, and technical specifications
- **Raw JSON**: Full datapackage.json content for technical inspection

## Example Structure

```
data/
├── input/                          # Default/legacy input folder
├── output/                         # Default/legacy output folder  
├── config/                         # Default/legacy config folder
├── nile-basin-analysis_a1b2c3d4/   # Case study 1
│   ├── input/
│   │   ├── water_demand.csv
│   │   ├── precipitation.csv
│   │   └── model_script.R
│   ├── config/
│   │   ├── model_config.yaml
│   │   └── parameters.json
│   ├── output/                     # Generated by GloWPa
│   │   ├── results.csv
│   │   ├── water_balance.png
│   │   └── execution.log
│   └── datapackage.json           # Case study metadata
└── drought-scenarios_b2c3d4e5/     # Case study 2
    ├── input/
    ├── config/
    ├── output/
    └── datapackage.json
```

This structure ensures that each water management project has its own isolated data environment while maintaining compatibility with existing workflows.
